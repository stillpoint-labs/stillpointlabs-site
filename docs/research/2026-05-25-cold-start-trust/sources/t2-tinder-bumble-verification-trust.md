## Key Findings

1. **Verification asks BOTH directions in the same step.** "Prove you are you" (to the platform) AND "the platform proves it has good intentions with that data" (encrypted face maps, named retention timeline, deletion guarantee). Both halves are necessary.
2. **The verification request is justified contextually.** Dating-app users understand the safety stakes; Tinder / Bumble explain WHY the verification is needed. Asking for the same disclosure in a context without obvious need would fail.
3. **The trust contract is bounded.** "We store this until you're inactive for 3 years, then delete" — naming the retention boundary is the trust signal. Indefinite retention destroys the same trust.
4. **The asymmetric collapse pattern reappears.** If verification data leaks, dating-app trust collapses across the category. The architectural choices (encryption, deletion timelines) exist because the platform CAN'T afford a leak.
